# Perfil de salida y diseño inverso de DAM/DAW

## 1. Idea central

**¿Qué queremos que nuestro alumnado sea capaz de hacer, demostrar y defender cuando salga del centro?**

A partir de esa respuesta se desarrollara el resto del proyecto: macrocompetencias, resultados de aprendizaje experimentales, situaciones de aprendizaje, tecnologías, metodologías, evaluación, colaboración con empresas y organización docente.

## 2. Modelo de diseño inverso

La secuencia propuesta es:

| Paso | Pregunta guía | Producto resultante |
| - | - | - |
| 1. Perfil de salida | ¿Qué debe saber hacer el alumnado al titular? | Perfil competencial acordado por el grupo. |
| 2. Evidencias | ¿Cómo sabremos que realmente sabe hacerlo? | Pruebas, productos, defensas, auditorías y portfolios. |
| 3. Macrocompetencias | ¿En qué grandes bloques se organiza ese perfil? | Líneas curriculares intermodulares. |
| 4. RA oficiales y experimentales | ¿Qué RA oficiales cubren esas competencias y que RA nuevos necesitamos probar? | Matriz de trazabilidad curricular. |
| 5. Situaciones de aprendizaje | ¿Qué experiencias profesionales permiten adquirirlas? | Situaciones HEXA conectadas con retos. |
| 6. Tecnologías | ¿Qué herramientas son necesarias para aprender y demostrar esas competencias? | Stack tecnológico mínimo, recomendado y avanzado. |
| 7. Evaluación | ¿Cómo ponderamos producto, proceso, comprensión, autonomía y colaboración? | Rubricas, listas de cotejo e instrumentos comunes. |
| 8. Validación externa | ¿Lo que proponemos encaja con empresas y CAPIFP? | Feedback y ajustes del piloto. |


## 3. Perfil de salida propuesto para debate

Este perfil es un borrador inicial. Debe ser discutido, priorizado y ajustado por el grupo de trabajo.

Al finalizar DAM/DAW en el IES Politécnico Hermenegildo Lanz, el alumnado debería ser capaz de:

| Ámbito | Competencia de salida |
| - | - |
| Resolución de problemas | Analizar una necesidad real, acotar el problema, formular requisitos, proponer soluciones viables y justificar decisiones. |
| Programación | Comprender, escribir, modificar, depurar y mantener código, tanto propio como generado o asistido por IA. |
| Arquitectura básica | Diseñar soluciones con separación de responsabilidades, interfaz, lógica, datos, seguridad y despliegue. |
| Backend y APIs | Crear servicios, endpoints, autenticación, validaciones, acceso a datos, gestión de errores y documentación técnica. |
| Frontend e interfaces | Construir interfaces usables, accesibles, responsivas y conectadas con servicios. |
| Datos | Modelar, consultar, proteger, transformar y presentar datos, incluyendo SQL, NoSQL introductorio, BI y calidad del dato. |
| Testing y calidad | Diseñar pruebas, ejecutar validaciones, detectar errores, registrar incidencias y mejorar el producto. |
| Seguridad | Aplicar buenas prácticas de autenticación, autorización, validación de entradas, protección de datos y configuración segura. |
| Despliegue | Preparar entornos, usar Git, documentar instalación, desplegar prototipos y resolver incidencias básicas. |
| IA responsable | Usar IA y agentes como apoyo profesional, con trazabilidad, validación humana, control de datos y defensa de decisiones. |
| Comunicación | Explicar requisitos, decisiones, errores, arquitectura, pruebas y resultados ante perfiles técnicos y no técnicos. |
| Trabajo en equipo | Colaborar con roles, acuerdos, tablero, revisiones, feedback y responsabilidad compartida. |
| Aprendizaje autónomo | Buscar información fiable, aprender herramientas nuevas, contrastar resultados y adaptarse a cambios tecnológicos. |


## 4. Niveles de prioridad

Para evitar que el perfil de salida sea inabarcable, se propone clasificar cada competencia en tres niveles.

| Nivel | Significado | Ejemplo |
| - | - | - |
| Imprescindible | Todo alumno titulado debería poder demostrarlo de forma autónoma. | Leer y depurar código, crear CRUD seguro, usar Git, defender una decisión técnica. |
| Recomendable | Debe aparecer en situaciones de aprendizaje, aunque con apoyo o alcance limitado. | CI/CD básico, dashboard, NoSQL, despliegue con contenedores, pruebas automatizadas. |
| Avanzado experimental | Competencias a pilotar en retos concretos o alumnado/equipos con mayor madurez. | RAG, agentes, MCP, AIOps, evaluación sistemática de IA, pipelines avanzados. |


## 5. Ejes de decisión para el grupo de trabajo

El grupo debería decidir que peso tendrá cada eje en el perfil de salida.

| Eje | Pregunta de decisión |
| - | - |
| Fundamentos | ¿Qué fundamentos no se pueden simplificar/automatizar aunque exista IA? |
| Sintaxis y mecánica | ¿Qué tareas repetitivas pueden perder peso para ganar tiempo en validación y arquitectura? |
| Tecnologías | ¿Qué stack mínimo común debe manejar todo el alumnado? |
| IA | ¿Qué usos de IA son obligatorios, permitidos, limitados o no aceptables? |
| Empresa | ¿Qué competencias esperan las empresas de un perfil junior realista? |
| Soft skills | ¿Qué competencias personales y profesionales se evaluaran de forma explicita? |
| Evaluación | ¿Qué evidencias demuestran mejor la competencia real? |


## 6. Propuesta inicial de macrocompetencias

Desde el perfil de salida se derivan estas macrocompetencias intermodulares:

| Macrocompetencia | Descripción |
| - | - |
| MC1. Análisis y especificación | Convertir necesidades en requisitos, criterios de aceptación, alcance y riesgos. |
| MC2. Desarrollo de producto software | Construir soluciones con frontend, backend, datos, seguridad y documentación. |
| MC3. Calidad, pruebas y auditoría | Verificar código, detectar errores, probar, revisar y mejorar. |
| MC4. Datos e inteligencia aplicada | Modelar, consultar, explotar y presentar datos; introducir BI, NoSQL y RAG básico. |
| MC5. Despliegue y operación básica | Usar Git, entornos reproducibles, despliegue controlado, logs e incidencias. |
| MC6. IA responsable y sistemas agénticos | Usar IA y agentes con control humano, trazabilidad, seguridad y validación. |
| MC7. Colaboración profesional | Trabajar por roles, comunicar, documentar, defender y reflexionar. |


## 7. Implicación curricular

Este enfoque permite que el proyecto de innovación explore una reorganización real del currículo:

| Acción curricular | Aplicación |
| - | - |
| Mantener | Fundamentos técnicos imprescindibles y RA oficiales vinculados al título. |
| Compactar | Tareas mecánicas o repetitivas cuando la IA permita desplazar el foco a comprensión y validación. |
| Reordenar | Aprendizajes de diferentes módulos para que aparezcan juntos en situaciones profesionales. |
| Ampliar | RA experimentales sobre IA, agentes, datos, seguridad, especificación, auditoría y despliegue. |
| Diferenciar | Mantener identidad DAM/DAW en productos finales, tecnologías y profundización. |


## 8. Próximo producto del grupo

El siguiente producto debería ser una **matriz de perfil de salida**, donde el grupo seleccione:

| Elemento | Decisión necesaria |
| - | - |
| Competencias imprescindibles | Las que todo alumno debe demostrar. |
| Competencias recomendables | Las que deben aparecer en retos o situaciones. |
| Competencias avanzadas | Las que se pilotaran de forma experimental. |
| Evidencias comunes | Las pruebas o productos que se usaran en todos los equipos. |
| Tecnologías mínimas | Herramientas que no pueden faltar. |
| Soft skills evaluables | Competencias personales que tendrán peso real. |
| Papel de IA | Normas de uso, limites y evidencias de trazabilidad. |
